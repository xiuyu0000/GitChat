# 持续研究系统 MVP 基座 - Repo Audit 报告

基于对 `GitChat` 仓库代码库（前端 React Flow, 后端 Express）的审查，以下是针对“构建持续研究系统 MVP”这一目标的评估与盘点。

## 一、项目现状理解

### 1. 当前项目的核心定位
当前项目是一个 **“受 Git 启发的非线性 LLM 聊天工具”**。它将传统的瀑布流式对话转换为了可视化节点与连线的形式，允许用户在任意对话节点进行分支复制和内容编辑。

### 2. 当前项目解决了什么问题
- 传统对话界面下，修改更早的对话会导致后续会话记录丢失。
- 难以直观地整合不同分支的对话上下文。
- 对于同一个 Prompt 的多次不同回复测试对比，在传统流中非常难以组织和对齐。

### 3. 项目本质分类（聊天/画布/研究）
目前它是一个带有 **画布操作（Canvas）的聊天工具**。
它具有基础的可视化布局，但**并没有达到“研究工具”的门槛**。它缺乏实体化的知识沉淀节点、证据检索流、明确的信息出处引用机制以及状态标记能力。

### 4. 核心交互模型
交互模型基于“图（Graph）”中的“前驱与后继”逻辑：
- 节点：分为 `UserInputNode` 和 `LLMResponseNode`。
- 边：代表上下文语境的继承。
- 数据流向：向 LLM 发送请求时，程序会**向上递归遍历目标节点的所有先驱节点**，重组出一条线性的聊天历史（Context）并发送。
- 级联更新（Cascade）：编辑某个早期节点后，系统会遍历它下面所有的后继 `llmResponse` 节点并依次重新请求。

## 二、代码与架构盘点

### 1. 技术栈
- **前端**: React 18, TailwindCSS, `@xyflow/react` (React Flow 核心引擎), `react-contexify`(右键菜单), `react-markdown`。构建工具为传统的 `create-react-app` (Webpack)。
- **后端**: Node.js, Express, `openai` (Node SDK)，目前通过代理配置接入 `OpenRouter`。

### 2. 目录结构
非常初级的两段式脚手架结构：
- `/nodechat`：CRA 生成的前端项目。代码主要集中在 `src/components/`。
- `/server`：单文件（`server.js`）Express 服务。

### 3. 入口文件
- **前端**: `src/App.js` 引用 `NodeChat.js`。`NodeChat.js` 是全应用唯一级别的重量级容器控制器（上帝组件）。
- **后端**: `server.js`，包含仅有的一个端点 `POST /generate`。

### 4. 关键模块与职责
- **`NodeChat.js`**: 控制节点的新增、删除、边连接，并处理画布所有的事件与状态。
- **`UserInputNode.js`**: 展示用户输入框，提供重生成的触发按钮（♻️），根据内容动态调整自身宽高。
- **`LLMResponseNode.js`**: 渲染 Markdown，提供编辑和折叠功能。
- **`Utility.js`**: 核心算法层。负责处理 SSE (Server-Sent Events) 的流解析，以及图结构的遍历寻找前驱/后置节点。

### 5. 数据流与状态管理方式
**极度耦合**。没有使用 Redux、Zustand 等状态库。所有状态直接硬编码存在 `NodeChat.js` 的 `useNodesState` 和 `useEdgesState` 中。组件通过调用 `reactFlow` hook 直接操作 DOM 元素和内部状态数组。位置计算甚至依赖了原生 DOM API（如 `element.offsetHeight`）。

### 6. 存储方式
**纯内存（Ephemeral）**。页面一刷新所有画布内容丢失。无任何本地持久化（LocalStorage/IndexedDB）及服务端持久化。

### 7. 外部依赖与第三方服务
- LLM API 通过 `OpenRouter` 调用 `anthropic/claude-opus-4-6`（**模型名称被硬编码死**）。

### 8. 当前测试情况
完全空白。仅存在 CRA 默认的 `App.test.js`。没有对应复杂业务（分支复制、级联生成、上下文组装）的单元测试。

### 9. 核心技术约束 (Constraints)
后端通过 `stream` 直接推送给前端，目前**无法中断或取消**请求。若发生误操作触发了包含 20 个节点的级联更新，在前端没有 UI 阻断、后端没有 cancellation token 的情况下，会面临算力消耗失控和展现逻辑错乱。

## 四、与目标产品的差距分析 (Gap Analysis)

要构成一个“持续研究系统 MVP”，当前项目存在明显的代差：

1. **产品能力缺口**
   - 目标系统不仅需要传统的“单向提问”，还需要具备**“思考能力”**（Thinking mode）以及“搜索、整理、总结”。
   - 缺少专属配置项：无法在全局和单个 `User Input` 节点上独立配置所选模型、思考模式、联网模式和专属 Prompt。这种细粒度的控制能力对对比研究至关重要。
   - 缺少专用研究节点：用于显示搜索回溯链接的节点（Citation Nodes）、阶段性研究结论提炼的节点（Summary Nodes）。
2. **交互缺口**
   - 一旦触发了级联生成，**无法中断请求**，这在研究场景由于 token/响应可能极长，会引起非常卡顿的问题。
   - 节点没有状态指示（如：Draft, Summarizing, Error, Done）。
   - 缺乏 Token 感知反馈：随着上下文化延伸，没有剩余 Token 数和当前可用 Token 比例的即时提示，极端场景下用户体验毫无预期保障。
3. **数据模型与展现缺口**
   - 图结构单纯承载文字聊天，缺少 Payload 规范化。对于系统的实体（文件内容、链接、多模态图文），没有统一的 Schema 设计支持。
   - 节点调整能力不足：在重度研究场景下，用户需要手动任意调整节点的大小（长宽高低）来进行排版与归类，目前缺乏自由拖拽改变尺寸的交互与机制。
4. **可扩展性缺口**
   - `NodeChat.js` 是意大利面条代码（Spaghetti code）。新增一个搜索节点意味着要在该文件内追加几十行样板映射逻辑。必须引入分层的状态机或类似 Zustand 的集中管理机制脱离 React 渲染层。
5. **未来架构阻力缺口**
   - 依靠计算 DOM (`offsetHeight`) 来决定新建节点落盘位置，当内容溢出、折叠或是使用小屏幕时，自动排列计算极为脆弱。

## 五、MVP 纠偏建议

针对“构建完整研究系统 MVP”，我给出以下独立判断和业务重塑建议：

### 1. 明确 MVP 必须具备的
- **坚守现有的图结构上下文提取机制**：这是本系统的灵魂。必须继续沿用当前“基于图节点连线的前置关系递归提取 Context”的大模型上下文管理方式，保持通过视觉分支梳理对话流的核心体验。
- **思考模式与联网能力的赋能**：系统不仅需要可以借助外力（网络搜索），需要依靠内力深度逻辑推演（思考能力）。
- **细粒度节点的模型与策略控制**：
  - 选择模型、控制思考模式、控制联网模式、并设定总体使用的提示词。这四个关键配置**不仅要加入到底部的全局输入框中，还必须作为核心属性加入到每一个单独的 `User Input` 节点对话框中**。这确保用户可以随时针对特异性分支做横向模型对比。
- **节点高低与长宽的自由调整能力**：允许用户任意缩放修改各个节点的长宽尺寸并随意摆放位置，这是在二维画布上规划与清理思路的基础操作。
- **持久化（自动保存）**：研究是持续时间的任务，丢失不可接受。建议在 MVP 通过 Dexie (IndexedDB) 先做强大的本地离线存储，后端存储可延后。
- **中断/取消生成（Cancellation）**：在重度思维探索中，随时丢弃某条推断分支是刚需。
- **Token 提示与分支溢出预警**：由于当前未规划上下文压缩功能，若无 Token 控制用户在研究后期将体验极差。这**决不是可选功能**。必须提供当前可用 Token 的计算与提示；**当某一分支的上下文累积到最大可用窗口的 60% 时，系统必须主动提醒用户“总结当前分支内容，并另起一个分支进行后续研究”**。
- **状态管理业务剥离**：必须将节点的 CRUD 剥离为统一纯粹的 Data Layer，确保“画布”仅仅作为“数据的可视映射”。

### 2. 可以且应当延后的
- 用户账号/多用户权限管控：作为 MVP，纯本地 Local First 足矣。
- 复杂的富文本深度导出机制：MVP 专注结构化 Markdown，无需立刻开发复杂的 Word 或 PDF 的图形渲染导出。

### 3. 未提起但必须补上（最小集合推荐）
- **只读分享或导出结构化格式 (Export)**：持续研究的终局往往是“产出一篇文章或报告”。画布本身非常难阅读，需要实现一个能将选定路径 (Path) 上的节点顺次合并导出为一层扁平 Markdown 的能力。
- **阶段标记与抽象节点（研究节点/摘要节点）**：对于复杂的多分支网络，非常需要对节点打 Label（如 `Hypothesis`, `Rejected`, `Conclusion`）。没有明确的高层抽象，面对大量对话气泡会导致知识无法收敛。

### 4. 可能会拖慢进度、应当暂缓的
- 绝对完善的“撤销/重做 (Undo/Redo)”：对于图模型实现时序性的 Undo 成本巨大，MVP 应暂缓，允许用户手动删除重新来过即可。
- 节点搜索/标签筛选：当前系统没形成长年累计的数据，通过浏览器原生机制可规避，暂缓。

## 六、风险判断 (Risk Assessment)

### 1. 最可能导致改造失败的 5 个关键风险
1. **状态管理爆炸**：无解耦的情况下直接加入更多节点逻辑与业务。
2. **Context Window 深度溢出**：缺乏合理的 Context 控制或无视递归上限，发版后导致调用被拒。
3. **API 级联雪崩（Cascade Avalanche）**：更改根节点引发非预期的上百次 LLM 并发，导致计费灾难。
4. **前端性能短板**：复杂页面下 React 流图帧率下降，特别是在长文本解析不断执行排版更新时。
5. **DOM 定位脆弱性**：继续沿用读取 `offsetHeight` 做高度依赖而带来的节点无限重叠错位事故。

### 2. 风险来源分解
- **来自原项目本身**：DOM 渲染依赖定位、缺乏 Cancellation 机制、强耦合状态数据、缺乏任何形式的错误重试和持久化。
- **来自新需求**：将网络流信息与已有可视化逻辑混装，导致复杂度上升；构建结构化思维导出工具时会遇到的解析难题。
- **需要先做小型验证的情况**：
  - “搜索节点”到“思维流”的融合。这个搜索模块如何吐出结构化上下文？
  - Zustand 结合 ReactFlow Headless state 抽取是否能真正改善维护体验（POC 验证）。
