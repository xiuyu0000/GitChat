# Continuous Research System MVP - Specification

## 一、背景与目标

### 1. 当前项目的原始能力
目前的 GitChat 仓库具备了一个“受 Git 启发的非线性聊天工具”雏形。它支持在一个基于 React Flow 的节点画布上，针对不同的对话气泡建立分支、修改提问，并自下而上回溯形成发给 LLM 的上下文。底层通过流式 API 返回结果并产生级联更新。

### 2. 为什么要把它演进为“持续研究系统”
传统的“一次性提问-回答”机制无法支撑复杂的思维作业。对于严肃的知识工作者而言，研究一个深层命题需要**海量信息试探、反复证伪、查阅外链资料、对比不同大模型的推理，并最终对大量碎片进行归纳提炼**。现有的工具要么只有线性流（丢失分支对比能力），要么仅是白板（缺乏模型对话流和自动回溯的能力）。我们希望保留 GitChat “可回溯、可分支”的图谱语境管理优势，将其重塑为专门帮助知识工作者“推进复杂智力劳动”的研究场域。

### 3. 这个 MVP 要解决的核心问题是什么
- **复杂上下文的收拢难题**：解决在超长对话路径中思维发散后难以汇聚、遗忘研究主线的痛点。
- **验证手段单一问题**：打破模型“幻觉”黑盒，通过融合“外部搜索结果”和“不同模型的异构推理”，为结论提供严谨支撑。
- **过程难持久化问题**：保障耗时数天甚至数周的研究工作得以“随时中停、随时恢复、安全落盘”。

### 4. 成功标准
- **跑通研究闭环**：用户可以在一套系统中完成“提问 -> （节点级）开启思考/联网 -> 模型对比 -> 摘录总结 -> 导出结果”的全流程，中间无需切换到原生浏览器查资料或切换到其他聊天工具。
- **稳定性达到可用级别**：在拥有单节点 2000 字、单图包含 50+ 节点的情况下，平稳运行无白屏，并保证刷新不丢数据。支持大语言模型流式输出的即时中断。

## 二、目标用户与场景

### 1. 目标用户是谁
研究员、独立开发者、重度知识图谱构建者、需要对比多种 AI 能力的产品经理、学术界师生，以及任何需要从事“超过单次提问就能解决”的脑力劳动者。

### 2. 典型使用场景
- **竞品/技术方案调研**：在根节点输入目标（如“Zustand 相比 Redux 的底层机制差异”），分别使用连线分化出“GPT-5.4 推理分支”与“Claude-4.7 带搜索分支”，在同一画布上比对它们的响应并在最后连结成一个总结节点。
- **小说/长篇剧本推演**：为同一情节分出 5 个走向（节点），让 LLM 续写后，废弃 4 个，顺着最优分支继续发展，并在旁边留下总结卡片。

### 3. 为什么他们不用普通聊天产品
1. **失去旁支对比**：普通聊天在某一节点回退重试后，被覆盖的那一条历史会丢失或者极难在同一视野里双屏对比。
2. **缺乏空间排版**：线性结构不允许用户将“前提证据”摆放至左侧，将“衍生思考”摆放到右侧进行分类管理。
3. **全局配置僵化**：普通聊天工具往往整个会话只能共享一套基础设定，无法在追问的第 8 个回合突然让系统采取另外一个专用于代码审查的 Model 和独立的系统 Prompt。

## 三、产品原则

1. **坚守图谱溯源，绝不妥协上下文组装逻辑**：节点的连线即代表上下文继承，必须保障可视化的链路等于真实提供给模型的 Context，所见即所构。
2. **研究过程透明且可验证**：思考链条和搜索信源必须是可视化的、独立的实体，确保知识出处有迹可循，拒绝信息黑盒。
3. **单点绝对自治**：每一个生成性动作节点都可以具备独立的人格、模型和工具选项（Model / Prompt / Search）。
4. **过程资产决不能丢**：研究是心智的高耗能活动，系统必须在每一个有效动作后静默保存，容忍用户的误操作或意外关闭。
5. **重聚合，轻发散**：鼓励分支发散，但系统机制必须为收敛服务（提供Token容量预警与分支强行建议，提供总结节点指引总结）。

## 四、MVP 范围

### 1. 画布基础能力
- **节点生命周期**：新增、双击编辑文本、选中删除。
- **排版操作**：随意拖拽移动、**任意拖拽调整节点的长宽/高低大小**以适应不同的研究信息量显示。
- **连线组织**：从 Handle 端点向另一端点拉线连接，也支持点击/选中虚线删除断开。
- **视口管理**：鼠标滚轮缩放、中键拖拽/空白处拖拽平移、框选多节点（依赖 React Flow 底层支持）。

### 2. 研究节点体系
我们将节点分为两类，**MVP 的最小有效节点集合如下**：

1. `User Node` (用户意图节点)
   - 包含用户的输入、角色定义。
   - 承载整个研究单步发起的**控制面板**（模型选取、思考开启、联网开启、重写提示词设定）。
2. `Generative Node` (模型生成节点)
   - 容纳 AI 助手输出的正文（Markdown）。
3. `Thinking / Tool-call Node Component` (内置思考与行动组件)
   - 网络搜索不作为独立的手动节点，而是依赖 LLM 的**自主 Tool Call** 机制。此过程中产生的思考（Thinking）与工具调用执行日志（Tool calls），统一作为附属结构“折叠卡片”，嵌入挂载在 `Generative Node` 内部（生成答案的前置位）。最大限度减小图谱视觉复杂度。
4. `Summary/Note Node` (研究沉淀节点)
   - 不参与 LLM Context 的回溯（切断上下文污染），仅作为用户自己在画布任意角落撰写备忘录、固定结论的文本块。

**【理由】**：若把“思考”和“搜索结果”全部做成独立在画布上卡着的连线节点，会导致整张图的连线极其冗长、结构不堪重负。将“搜索”（由大模型自主判定的 Tool Call）和“思考”展示降级为嵌在 `Generative Node` 内的过程折叠面板，不仅完美适配前端组件呈现，也能避免破坏图谱核心的对话逻辑继承关系。独立的 `Summary Node` 则能解决研究梳理时缺少独立白板备忘录的痛点。

### 3. 对话问答能力
- **基础推演**：沿用原带状上下文拼接方式和普通的 Markdown 流式渲染。
- **能力开关与干预**：支持在全局以及单个 `User Node` 级直接配置选项开启思考模式、开启联网能力。
- **自定义环境**：利用统一的后端代理服务（如设置好配置的 OpenRouter）中继多模型请求。在前端的全局或各个 `User Node` 中直接通过清单下拉即可切换模型（无须在各处繁琐配置不同的 API Key），同时允许复写系统 Prompt。
- **可用性保障**：Token 生成的实时更新、 Token 总量预警机制以及极速的中断按钮（Cancellation）。

### 4. 多画布管理
- **工作区会话**：提供列表 UI，支持多 Session 画布的创建、重命名、删除。
- **秒级切换**：无后端重度握手，基于纯本地加载的极致画布切换速度。
- **全静默自动保存**：基于 IndexedDB (如 Dexie.js)，每一次节点位置变动、文字失焦、模型响应结束都会刻意覆盖当前会话对应的快照。不需要手动点击“保存”。

### 5. 导出能力
**【强力推荐支持】：单分支结构化 Markdown 导出。**
- **理由**：作为研究系统 MVP，导出能力是闭环核心。画布很难被团队其他同事直接阅读，因此必须选择选中终端节点后，顺着父节点反向合并为一篇扁平视角的 Markdown。这种格式是对各类外部工具（如 Notion, Obsidian）亲和度最高的纯文档格式，远胜复杂的 JSON 或难以操作排版的 PDF/图片导出。

## 五、必须补充的 MVP 基础能力

在原需求以外，必须强行塞入 MVP 的兜底型工作，防止系统体验“崩塌”：

### 1. Must Have for MVP
- **Token 使用提示与 60% 容量分支截断预警**
  - **为何必须**：目前系统不做上下文压缩算法，研究深入后深度分支容随时触达 128k / 200k 限额死机。
  - **表现**：计算每条线的当前可用及当前占用 Token；**当单分支 Context Size 到达容量上限的 60% 时，必须通过 UI 强硬提示预警：建议用户总结当前分支内容，另起新分支研究**。
- **流生成全链路的中断 (Abort Request)**
  - **为何必须**：级联更新往往非常消耗计算与计费资源，若发现回答走向偏离需及时打断。
  - **取消表现与策略**：当尝试修改并重生成一条很深的链路受到人为截断时，所有处于排队下游尚未被处理的级联子节点，**必须保持历史原样**，绝不可破坏性清空转为错误空白状态。
- **基本的失败提示与错误恢复 (Error States UI)**
  - **为何必须**：由于依赖网络极其明显，Token 耗尽、无网络、代理服务崩溃时，若没错误状态红框标注在故障阶段节点上，整张图将变成无效的不可解释状态。

### 2. Nice to Have after MVP
- **撤销/重做 (Undo/Redo)**：多重并行的状态快照在 IndexedDB 中的回滚成本过高，对于初期版本过于沉重。
- **高级的来源引用标注 (Citation Visualization)**：MVP 可以把 Search 的 URL 附带在下方作为纯文本锚点结构即可，复杂的双向溯源高亮先放以放。
- **节点标签与结构化搜索 (Tags / Node Search)**：当内容极多时有价值，初期在可承受图幅内先靠缩放平移以及浏览器自带的 Ctrl+F 纯文搜索即可应对。

## 六、非目标 (Non-Goals / Out of Scope)

本阶段 MVP **绝不投入做以下行为**，即便其听起来非常诱人：
1. **实时多人协作 / 复杂权限系统**：不做 Figma 级带实时光标指针追踪的群智协同协作。纯粹聚焦于单人的 Local First 工作流。
2. **集中化或重量级云端知识库接入**：不接所谓的各种基于业务自身的文档切片私域 RAG；网查纯指向大众互联网 API（例如基于 DuckDuckGo 的中继）。
3. **高级工作流编排**：这不是一个 Coze 或 Dify，我们的图不承担把模型 A 生成 JSON 给模型 B 这种“数据编排管道”逻辑，仅作为用户的交互空间展示层。

## 七、功能需求 (Functional Requirements)

| 编号 | 模块 | 需求描述 | 验证标准 |
| :--- | :--- | :--- | :--- |
| **FR-01** | 画布存储与会话 | 支持利用 IndexedDB 创建、切换和删除多个独立画布。节点编辑、拖拽和连线状态全生命周期由本地静默持久化。 | 打开并退出应用多次，修改过的节点位置与尺寸像素不差，无需手动点击 Save。 |
| **FR-02** | 连线与溯源基础 | 基于现有的有向边结构维护上下级血缘，任意新生成操作从当前触发展开沿着祖先向上递归收集 Payload 形成长串上下文。 | 上下文生成计算 100% 稳定不出环（DAG安全）；最终送入大模型的序列有绝对正确的 User/Assistant 顺序。 |
| **FR-03** | 节点自由尺寸排版 | 各类渲染节点边框附带 Resize 控制柄，使用者通过拖拽边缘即可无极随意拉长拉宽图块高度和宽度。 | 长文可自动根据手动调节的框宽进行折行；自定义的宽高数据刷新不丢失。 |
| **FR-04** | 节点精细控制盘 | 每个全局输入框与单独建立的 `User Input` 节点框都内置一个配置悬浮盘，允许独立调配：①所用模型 ②是否启动思考 ③ 是否联网搜索 ④ 重写提示词。 | 针对同一祖先生成的不同子节点，观察后端可查出向 OpenRouter 分发不同模型参数/联网设定的标志位。 |
| **FR-05** | 推发级联控制与中止 | 点击再生或者确认按键后流式输出，并且暴露“取消”按钮支持主动挂断。若修改后引发了向下递归更新，中途取消时未重绘的节点需维持旧状。 | 触发中止后，该 Socket `Abort` 控制台显现截断；后续未轮到的排队节点状态完整，历史信息无损遗留。 |
| **FR-06** | 附属动作的折叠呈现 | 对于启用了思考（Thinking）和联网（Search）得到的额外推理日志、URL链接清单，作为一段折叠/分块的 Payload 显示嵌在对应 `Generative Node` 主文上方。 | 保证视图界面的高降噪率，确保大模型生成的干货与前置冗长处理日志相互区隔，收放自如。 |
| **FR-07** | 主动防过载机制 | 引擎监控每次发包前的上下文长度量（近似 Token），不仅显性显示且一旦突破当前使用限制的 60% 后，UI 侧强醒目弹出提示促使用户打断重开链路。 | 在累计发送数万字内容后必定触发此预警模块；不会意外静音被限流卡死。 |
| **FR-08** | 研究随笔节点 (Summary) | 提供画布独立生成的“笔记标签”（纯文本 Markdown 框），可以放置在各处作思路纪要。 | 放入笔记节点，验证该节点内容并没有污染随后发往 LLM 的核心前驱继承上下文内。 |
| **FR-09** | Markdown 纯化导出 | 右键任意单点选择“导出该链路记录”，沿着它追溯至根节点并在本地形成一个附带 Role Heading 结构的直读 `.md` 文本下载至浏览器。 | 在多分支庞杂图谱中选中一片叶子导出的绝对是一条完整连贯且排除无用旁支死胡同的信息干道文本。 |

## 八、非功能需求 (NFR)

- **性能 (Performance)**：图谱单张图支持 50+ 以上繁重 Markdown/富文本渲染的节点平滑铺展、缩放（FPS > 45），没有过度严重的不必要全量组件重绘 (React Re-renders)。
- **稳定性 (Stability)**：网络代理或 SSE Stream 中断、异常 JSON 解析时能被全局 Exception Controller 截获，直接反馈报错组件 UI 告知用户而不引发界面白屏锁死。
- **可恢复性 (Recoverability)**：IndexedDB 失效时应具备回退策略并尝试 LocalStorage 或 Memory 降级警告，保证不毁坏原本图谱的现有缓存序列化格式。
- **易用性 (Usability)**：节点连线吸附热区要对非开发者友好，缩放必须跟随主操作聚焦，且空状态进入系统需要有明确的文字引导建立根极 User 节点。
- **可维护性 (Maintainability)**：废弃原本使用真实 DOM 元素 `offsetHeight` 和状态全包混写的面条代码，使用 Redux/Zustand 这样的层抽离为完全去 UI 化的 Headless 图状态库。
- **可测试性 (Testability)**：类似上下文遍历拼接算法、Token 计算报警阈值校验、JSON 还原引擎这些核心函数必须写成 Pure Function，无需在沙盒 DOM 测试环境即可直接利用 Jest/Vite 撰写覆盖率单元测试。

## 九、约束与依赖 (Constraints & Dependencies)

1. **当前代码与状态层约束**：
   目前的 `App.js` `NodeChat.js` 是意大利面代码，状态重度嵌合 React Flow Hook 且依赖 DOM Api 判断坐标，这是最大死穴。由于此历史债，必须在 MVP 执行之初腾挪一定的期数做 Zustand 化重构；否则无法承载新 `Summary Node` 等组件。
2. **大本营存储机制**：
   因为完全重度依赖 IndexedDB API 取消了数据库，任何运行该程序的无痕窗口（Incognito）都会在关闭后完全丢失所有资产。这点对用户的心智可能带来挑战，必须依赖客户端明确的提示来打消隐患。
3. **接口扩展网关 (OpenRouter / APIs)**：
   假定 MVP 可以利用统一代理（例如 OpenRouter 等）消除针对 OpenAI/Anthropic 逐个接底层 API SDK 处理思考/返回数据不统一等复杂接口层。如果做原始 API 拆包需要多支付大量后端代码开发代价。

## 十、验收标准 (Acceptance Criteria)

以下提供核心模块的任务拆解验收视角：

- **视图与画布底层重构**
  - 使用者能框选、任意长宽调节节点布局并得到精准记录，刷新不断连、位置和内容一致性通过验证。
- **图状态抽离（Data Layer）**
  - 代码 `NodeChat.js` 从一个超 900 行的神级组件拆解成独立控制数据流动（Zustand Store）、连线策略与 UI Renderer ，图的结构数据与渲染严格分离。
- **研究对话全链路流转**
  - 能在长文本研究末端点开启“带联网的 Claude”与同样端点开启“带 Thinking 的 GPT”，通过不同模型进行级联。最终发往服务端的 Proxy API 参数被挂带明确。
- **防灾与防错机制跑通**
  - 一边生成一边拔掉网线，错误能通过边框标红等明确告诉用户。
  - 生成长内容过半立刻点 Abort 制止，当前生成废弃打平截断停止，浏览器不会继续阻塞并转圈。
  - Token 超 60% （或者自己设定个 50 token 小数量门槛模拟测试）后，预警器显眼弹窗或长挂在主干链路显眼位置。

## 十一、Assumptions & 产品迭代路线图

### 1. Assumptions (核心开发前提与已知决策)
- **多模型统一代理接入**：明确接入如 OpenRouter 等统一的后端网关。MVP 阶段只需利用后端 `.env` 配置一个全能网口即可下发与控制前端可选模型列表，不需在前端实现类似配置专属大厂底层独立接口的复杂弹盘。
- **搜网与工具执行逻辑**：明确放弃手动设立单一独立的“Search Node”，全面转向遵循大语言模型**自主触发的 Tool Calling** 方案。大模型基于工具返回执行二次组织思考，检索日志仅以折叠式信息嵌于问答生成气泡内。
- **安全中止不毁损体验**：在修改长辈节点引发的级联重渲染队伍中，任何针对当前动作或全链动作的突然中止（Cancel），其排队靠后的子孙节点数据将被停止覆写，并确保**必须留存历史原装数据**不致于转为报错空节点状态。
- **环境预期**：目前系统仅面向桌面级大显示器（PC）工作排版，当前不在手机移动端做高密长画板的排布降级妥协。

### 2. 产品迭代路线图 (Roadmap)
为确保 MVP (V1) 达到最快速的产品逻辑闭环验证，并清晰呈现此项系统未来扩展张力，确认以下各期研发边界：

**【V1 阶段：最小可用研究框架 (MVP) - 当前执行核心】**
- **（核心重构）**：React Flow 视图层脱离与 Zustand 驱动的 Headless State Data Layer 重写。
- **（持续存储）**：IndexedDB 零感知的原位高密存储保护。
- **（节点增强）**：所有用户输入节点的属性解耦独立（可任意定义当前所选的模型、思考状态、网搜启用、提示词设定）。
- **（可用保障）**：任意调整元素大小比例排布版图；提供上下文容积（Token量）60%到达时的强制切分建议提示环；实现即时的取消截断控制流。

**【V2 阶段：语义化拓扑图网络与高级校验阶段】**
- **边缘连接语义化（Edge Semantic）**：将单一的“时空继承连线”系统性升级为“带有属性意图设定的有向标注图”。不仅允许建立连接，更允许用户标注出边的语义：“因为 A，导致 B”或“用该资料佐证了假说 C”。支持依据不同连线颜色梳理不同门类的思想脉络体系验证工作。
- **长级联刷新的交互拦截增强**：如果研究者修改的是带有海量衍生图结构的“始祖级根节点根基错字”，将提供二次防卫弹窗阻断，区隔开“仅重写本次笔记”、“手动全量强迫级联衍生节点更新”。
- 节点多维度 Tags 标记系统与全局跨链关键字筛选高亮体系。

**【V3 阶段：文件解析多模态接入与高价值企业私库连接】**
- 连线画布增加强力多模态外挂接支持（随手拖入结构化图片引用；解析上传复杂的 PDF 后段落直接成为引用基石证据节点）。
- 检索行为库从通识公众互联网定向引入、结合高密级的企业专有知识库（RAG API）。构建私密研发环境内的专有论断依据中心。
