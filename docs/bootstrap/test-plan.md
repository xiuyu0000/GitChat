# Continuous Research System MVP - Test Plan

基于 `repo-audit.md` (代码审计), `spec-draft.md` (规格定义) 和 `design-draft.md` (架构设计)，本文档致力于设计一套具备实施性、立足于现有代码缺陷现状并面向最终高质量交付验收的测试计划。

---

## 一、测试目标

### 1. 本次 MVP 要验证哪些核心能力
- **数据一致性验证**：图谱的节点位置、连线、内容结构在页面刷新与会话切换中绝不丢失。
- **算法精准度验证**：对于任意深度的网状（DAG有向无环图）结构，其上下游 Context 回溯提取 100% 符合语义路径。
- **容错与防灾验证**：强行中止大模型回答、网络异常断开等情况下，UI 不白屏死锁，并正确挂载 “未对齐 (Stale)” 警报等安全提示。
- **容量监控验证**：当极高密度的研究文字出现时，Token 容量 60% 阈值警报能稳定抢先拦截。

### 2. 测试总体目标
在不盲目追求 100% 覆盖率的前提下，剥离原来耦合在 React Component 中的脏逻辑，围绕“研究状态安全流转”完成最核心纯函数与状态机的自动化覆盖，**避免将严重破坏用户知识库的低级 Bug 带到生产环境**。

### 3. 成功标准
- 核心图编排算法单元测试覆盖率 > 95%。
- IndexedDB 数据 Hydrate 与 Zustand Store 状态机链路完成主线用例集成测试。
- 完成关键手动探索测试矩阵验收，无 Crash/数据损坏/死循环。

## 二、测试范围 (Scope)

### **In Scope (本期重点验证)**
- 数据模型核心（图遍历、状态机切片）。
- `GraphContextBuilder` (提取机制) 纯函数化业务验证。
- Token 阈值告警机制触发测试。
- Abort Cancellation (立即中止网络并在排队节点实施降级留存处理) 测试。
- IndexedDB 持久化链路闭环测试。
- 单条分支链路 Markdown 提取生成完整性测试。

### **Out of Scope (不作为 MVP 测试阻塞项)**
- 视觉排版样式的跨浏览器/移动端像素级兼容性测试（系统定位面向桌面 PC 且不在移动端应用排版）。
- 大模型本身回答效果质量的准确率评估（模型质量非平台系统 BUG）。
- Tool Calling 的复杂网络穿透验证（测试中均以静态 Mock Payload 作为外部网络能力的返回数据标准）。

## 三、测试开发同频策略 (TDD & Testing Layers)

为了最高效地缩短问题排查时间并保证架构质量，**严禁将测试堆积到功能开发尾期进行**。所有特性开发过程必须严格贯彻 **Test-Driven Development (TDD) 同频策略**：随走随测。

- **Unit Tests (单元测试)**: 业务代码改造前率先落笔。针对无副作用的 Pure Functions 兜底。如开发 Phase 0 重构时，率先用 Jest 锁定 `GraphContextBuilder` (遍历算法)。
- **Integration Tests (集成测试)**: 伴随持久化（Phase 3）业务同步推进。专门校验 Zustand Headless 层与 Dexie.js 之间防抖刷入 IndexDB 表的一致性。
- **End-to-End Tests (E2E测试)**: 紧跟高危 UI 特性（Phase 2/4）研发。自动化模拟长链生成，触发全局中止 (Abort) 按钮，强制验证页面能精确给子节点打上红色 Stale 警报标识。
- **Manual Exploratory Tests (手工探索性测试)**: 仅在各项特性 Merge 前执行体验摸底。专门捕捉极限拉伸节点变型、节点重叠等极难通过代码自动判定优劣的视觉瑕疵。

## 四/五、按功能模块输出测试方案

### 1. 画布基础操作与界面
- **测试点**：节点 CRUD与排版拉伸。
- **正常路径**：双击空白生成新节点 -> 拖拉四周边角控制将其拉长 -> 刷新页面，节点大小坐标记忆无损。
- **边界条件**：把节点宽高拉到一个极小值，或拖到视口非常边缘。
- **异常路径**：将 A 连向 B，又将 B 尝试强行连回 A（制造死锁环）。系统必须在连线释放瞬间通过 `verifyConnection` 拦截并阻断该边。
- **Mock/自动化**：可使用 Playwright 进行 E2E 拖拽自动化，优先人工验证。

### 2. 对话问答与智能推演
- **测试点**：Token 计数、多模型选择分流、被中止生成的未对齐警告 (Stale Tag)。
- **正常路径**：修改爷爷节点发问内容，点击再生成。子节点顺序更新并加载新文。
- **异常路径 (Abort)**：同上，若有 5 个后继节点被排队等待生成，而在第 1 个生成过半时点击 Cancel。
- **状态一致性**：剩下的 4 个节点数据不可删除，UI 上必须给这 4 个节点显性打上“未对齐 (Stale)”的报错红角标。
- **依赖 Mock**：Mock 一个延迟响应的 SSE Server 接口用于故意拖长生成时间便于点击 Abort，优先级 **Highest** (E2E可自动化)。

### 3. 多画布管理与持久化 (IndexedDB)
- **测试点**：多 Session 独立运作；断电重开不丢字。
- **边界条件**：在一张画板产生超大内容（10万字）并伴随多次编辑时，保证 Debounce 500ms 不锁死主进程跳帧。
- **异常路径**：禁用 IndexedDB 的浏览器隐身模式启动系统，UI 需全局发红底警告用户“未开启持久化支持”。
- **自动化优先级**：Integration (优先利用 fake-indexeddb 与 Zustand 测试同步逻辑)。

### 4. 数据导出闭环
- **测试点**：单线逆向导出 Markdown；全库 JSON 导出与恢复。
- **正常路径**：在含有多个不相干死胡同的图上选中一个成功节点，点击导出。产出仅包含这条路径从头到尾干净对口的 `.md` 文件。
- **异常路径**：导入错误的残破的垃圾 `.json` 项目文件，系统捕捉异常并不覆盖当前缓存。

## 六、高风险测试专题

必须专门防卫的基础高危区：

1. **多模型/联网返回结果不一致，如何设计稳定测试？**
   - **策略**: 不依赖外部真网。建立一个 `msw (Mock Service Worker)`，在其 Interceptor 路由中硬编码配置。针对特定输入“测试Thinking”指令，固定向 UI 推送拥有 `<tool_calls>[Search]</tool_calls>` 结构假数据的 Stream 进行组件渲染还原性测试。
2. **级联更新/再生成时，如何验证不会破坏节点关系？**
   - **策略**: 建立有 10 个孙子节点的树，并针对根部执行深度的 Zustand Store 级联重试触发测试。断言重试后，被抛弃并抹除的老节点数量、替换新增的新节点 UUID 是否完美符合 10 个的数量守恒（不应出现孤儿节点溢出图外）。
3. **导出时如何验证顺序与引用完整？**
   - **策略**: 设计一份包含三个层级对话（且带有错误冗余连接）的 `mock-graph.json`。通过 `MarkdownExporter` 纯函数执行提取。利用 Jest 将产出文本同我们人工预设好的 `.md` `Golden File (Snapshot)` 测试进行严格字符核对校验。

## 九、自动化测试建议

### 1. 部署切分建议
- **Unit Unit**: 优先对算法层出兵（Context 遍历、Token 定制法、Markdown 组装函数）。
- **Integration**: 测试 Zustand Store Action 发起时是否准确将 Diff 发出传递给了 IndexedDB 持久化存盘方法。
- **E2E / 人工**: `react-flow` 面板节点尺寸鼠标拉长拉宽、长短线链接错乱问题必须交给人工验收。由于前端环境重绘频繁，使用 Playwright 处理部分诸如“能否成功弹出模型配置悬浮窗”这类死板回归测试即可。

### 2. 当前代码所需的前置重构测试代价
- **在当前老架构体系下，可测试性为零。** 因为核心的数据全被闭包封装在了一个庞大 React `App/NodeChat` 的 State 流下，且强依赖了真实 DOM 长宽高做渲染。
- **要进行测试重构必须**：执行架构改造方案 Phase 0，即分离数据大动脉为独立的 `Zustand` 和 `GraphContextBuilder.ts`。

## 十、回归测试清单

针对后续迭代（如做做改改产生牵连 BUG），设定局部动土强制回归范围：

1. **若改动画布引擎或缩放排版等拖拽逻辑**：
   - 必须回归验证：拖拽会不会触发全屏重新发送大模型网络 API 重求；长文本极速拖拽时图框高度是否自动适配，是否造成节点覆盖重叠。
2. **若改动模型/检索等 Tool-Calling 特性解析规则**：
   - 必须回归验证：折叠面板能否正常收拢含有换行或错误 Markdown 标签的残废标签；能否让其不污染纯正文部分的 Markdown 导出。
3. **若改动节点基础逻辑时**：
   - 必须全网回归验证：撤销动作能否安全执行，修改爷爷节点进行重新推演时，连同下游子代的祖宗体系关联线是否没被剪报作废。

## 十一、质量门槛 (Quality Gates)

| 门槛卡面 | 目标界定 | 拦截标准 |
| :--- | :--- | :--- |
| **本地单元开发门槛** | `Context`遍历、`Store`流转等纯功能逻辑 | 所有 Pure Logic 层行级/分支级 Jest 测试覆盖率不能低于 90%。 |
| **CI 自动通过门槛** | Push Commit 或打 PR 发起门槛 | 无严重 TypeScript 类型越界报错，单元防线测试绿色 ALL PASS。 |
| **发布 (MVP Deliver) 门槛** | 面向交付使用的实操 | E2E (Cancel / 导入导出落盘 / 阈值弹窗) 测试链全通过；人工测试满跑高并发级联节点 50 个不宕机、刷新位置无偏移。 |
| **可带着发布的小毛病** | 可以留待以后解决的技术债务 | 小范围无恶性后果的连线折叠重合（UI 绘制渲染不优美）、导出文件格式未做深度润色美化等体验小瑕疵。 |
| **绝不能发布带出门的** | P0 一票否决性质地红线 | 误触引发级联生成狂跑消耗千字 Tokens 而无法点击“立即取消”按钮导致崩溃的。导出内容存在残缺图或偷换上下文导致的“知识毒化伪造”。 |

## 十二、当前代码测试成熟度评估 (Assessment)

1. **现有测试是否足够**：**几乎为 0**。典型的原型项目，仅有 Create-React-App 自带默认骨架。
2. **最难测试区**：依赖使用 `offsetHeight` 的 DOM 获取自动规排旧机制。脱离浏览器甚至渲染流水线其根本返回不了结果。
3. **最值得优先补测区**：`GraphContextBuilder` (旧 `Utility.js`)。图连错一根线，LLM 上下文就错了整盘底稿。
4. **走向发布的最小同步测试建设路线 (TDD 视角)**：
   - **伴随 Phase 0**：编写 `GraphContextBuilder.test.ts` 强断言图遍历算法无瑕疵（无需 React）。
   - **伴随 Phase 3**：起草 `zustandStore.test.ts`，脱开 UI 强灌 30 次高频变更验证 IDB 防抖落盘完美响应。
   - **伴随 Phase 2/4**：利用 MSW 延时响应打通 Cancel (Abort) 断点和 Stale 报错角标的渲染 E2E。
   - **严守交付底线**：绝不把补测试当“事后擦屁股”的任务，测试脚本本身即是功能可用的证明人。

## 十三、已知前提与需决断区 (Assumptions & Open Questions)

- **Assumptions**: 考虑到研究画布工具具有较高的使用门槛，并假设在非低端浏览器容器设备中使用；性能测试部分基准锚点设定在我们不低于桌面端配置运行 Chrome/Edge，且 `fake-indexeddb` 具备足够能力仿真测试数据库写盘无延迟误差。
- **Open Questions**: 我们是否需要在集成 CI 流水线中真的分配一定的固定余额用于远端调用测试（比如接一个最低价位的智障开源小模型供打通网络全环节用），还是要求团队开发在此阶段 100% 坚持依靠静态网络 Mock 数据做流解析欺骗判断（防范资金与网络阻塞导致的假性挂断错误）？
